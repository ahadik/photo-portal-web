# Photo Portal Cloud Functions

This directory contains Firebase Cloud Functions (2nd generation) that handle server-side processing for the Photo Portal application. These functions run on Google Cloud and are triggered by events or HTTP requests.

## Overview

The Cloud Functions in this directory provide the backend processing layer for Photo Portal:

- **Image Processing**: Automatically processes uploaded photos (resize, extract metadata, generate thumbnails)
- **Geocoding**: Converts GPS coordinates to human-readable location names
- **Message Management**: Handles sending photo messages from admins to the device
- **Data Management**: Maintains JSON index files (`photos.json`, `messages.json`) in Firebase Storage

## Functions

### `processUpload`

**Type**: Storage-triggered function (2nd generation)  
**Trigger**: Automatically runs when a file is uploaded to `photo-portal-media/uploads/`  
**Purpose**: Processes raw photo uploads from the admin portal

**What it does**:
1. Downloads the uploaded image from Firebase Storage
2. Extracts EXIF metadata (capture date/time, GPS coordinates, orientation)
3. Corrects image orientation based on EXIF data
4. Resizes the image to fit within 2048×2048 pixels (preserving aspect ratio)
5. Generates a 400px-wide thumbnail
6. If GPS coordinates are present, calls Mapbox Geocoding API to get location name
7. Uploads processed image to `photo-portal-media/photos/{photoId}.jpg`
8. Uploads thumbnail to `photo-portal-media/thumbs/{photoId}.jpg`
9. Updates `photos.json` in `photo-portal-data` bucket with new photo metadata
10. Deletes the original upload from `uploads/` folder

**Configuration**:
- Memory: 1GB (for image processing)
- Timeout: 120 seconds
- Trigger bucket: `photo-portal-media`
- Trigger path: `uploads/**`

### `sendMessage`

**Type**: Callable HTTP function (2nd generation)  
**Trigger**: Called from the admin portal via Firebase SDK  
**Purpose**: Creates a photo message that will be displayed on the device

**What it does**:
1. Verifies the caller is authenticated (Firebase Auth)
2. Validates the request (photoId exists, message text is 1-280 characters)
3. Verifies the photo exists in `photos.json`
4. Creates a new message entry with unique ID
5. Appends the message to `messages.json` in `photo-portal-data` bucket
6. Returns the created message object

**Authentication**: Requires Firebase Auth token (admin account)

**Request format**:
```typescript
{
  photoId: string;  // ID of photo to attach message to
  text: string;     // Message text (1-280 characters)
}
```

**Response format**:
```typescript
{
  id: string;       // Unique message ID
  photoId: string;  // Photo ID
  text: string;     // Message text
  sentAt: string;   // ISO timestamp
}
```

## Prerequisites

- **Node.js 24** (specified in parent `.nvmrc`, use `nvm use` from project root)

## Setup

1. **Navigate to the functions directory**:
   ```bash
   cd functions
   ```

2. **Install dependencies**:
   ```bash
   npm install
   ```

3. **Configure Firebase** (if not already done from project root):
   ```bash
   firebase login
   firebase use --add  # Select your Firebase project
   ```

4. **Set environment variables/secrets**:
   
   For local development, create a `.env` file (not committed to git):
   ```env
   MAPBOX_TOKEN=your_mapbox_token_here
   MEDIA_BUCKET=photo-portal-media
   DATA_BUCKET=photo-portal-data
   ```
   
   For production deployment, set secrets via Firebase CLI:
   ```bash
   firebase functions:secrets:set MAPBOX_TOKEN
   ```
   (Enter your Mapbox token when prompted)

## Configuration

### Environment Variables

Functions use the following environment variables:

- **`MAPBOX_TOKEN`**: Mapbox API token for reverse geocoding (required for location names)
- **`MEDIA_BUCKET`**: Firebase Storage bucket name for media files (default: `photo-portal-media`)
- **`DATA_BUCKET`**: Firebase Storage bucket name for JSON data (default: `photo-portal-data`)

### Setting Secrets in Production

Use Firebase Functions secrets (recommended for sensitive data):
```bash
firebase functions:secrets:set MAPBOX_TOKEN
```

Access in code via `process.env.MAPBOX_TOKEN`.

### Function Configuration

Function-specific settings are in `src/index.ts`:

- **`processUpload`**:
  - `memory: "1GiB"` - Memory allocation (needed for image processing)
  - `timeoutSeconds: 120` - Maximum execution time
  - `bucket: MEDIA_BUCKET` - Which bucket to monitor

- **`sendMessage`**:
  - `enforceAppCheck: false` - App Check not required (can be enabled later)

## How It Works

### Image Processing Flow

1. Admin uploads photo via admin portal → file goes to `photo-portal-media/uploads/{uuid}.jpg`
2. `processUpload` function automatically triggers
3. Function downloads, processes, and uploads results
4. Function updates `photos.json` with new photo metadata
5. Device app polls `photos.json` and sees new photo
6. Device downloads and displays the photo

### Message Flow

1. Admin selects photo and writes message in admin portal
2. Admin portal calls `sendMessage` Cloud Function via Firebase SDK
3. Function validates request and creates message entry
4. Function updates `messages.json` with new message
5. Device app polls `messages.json` and detects new unread message
6. Device shows message indicator LED and displays message when button pressed

## Troubleshooting

### Build Errors

**TypeScript errors**:
```bash
npm run build
```
Check the error messages. Common issues:
- Missing type definitions: Install `@types/{package-name}`
- Import errors: Check file paths and exports

**Missing dependencies**:
```bash
npm install
```

### Deployment Errors

**Authentication errors**:
```bash
firebase login
```

**Permission errors**:
- Ensure your Firebase project has Functions enabled
- Check that you have the correct project selected: `firebase use`

**Function timeout**:
- Increase `timeoutSeconds` in function configuration
- Check if image processing is taking too long (very large images)

### Runtime Errors

**Mapbox geocoding fails**:
- Verify `MAPBOX_TOKEN` is set correctly
- Check Mapbox API quota/limits
- Function will continue without location name if geocoding fails

**Storage access errors**:
- Verify bucket names are correct
- Check IAM permissions for Cloud Functions service account
- Ensure buckets exist in Firebase Storage

**Image processing errors**:
- Check that uploaded file is a valid image
- Verify `sharp` library is working (may need system dependencies on some platforms)
- Check memory allocation (1GB should be sufficient for most images)

### Viewing Logs

**Recent logs**:
```bash
firebase functions:log
```

**Specific function**:
```bash
firebase functions:log --only processUpload
```

**Real-time logs**:
```bash
firebase functions:log --follow
```

## Best Practices

1. **Always test locally first** using emulators before deploying
2. **Check logs** after deployment to ensure functions are working
3. **Monitor function execution time** - optimize if approaching timeout
4. **Set up alerts** in Firebase Console for function errors
5. **Keep dependencies updated** - run `npm outdated` periodically
6. **Use TypeScript strictly** - fix all type errors before deploying
7. **Version control** - commit `package.json` and `package-lock.json`, but not `lib/` or `node_modules/`
